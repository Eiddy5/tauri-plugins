# macOS 同应用多窗口浮层遮挡设计

## 状态

已于 2026-07-14 确认采用“按角遮挡、拖动隐藏、松手重算”方案，优先兼容 ToDesk 等同进程多窗口应用。

## 问题

窗口共享使用四个独立 `NSPanel` 绘制目标窗口四角边框。对于普通单窗口应用，`orderWindow:relativeTo:` 可以把四个 panel 连续排在目标窗口正上方；对于 ToDesk 这类同一进程同时存在多个 layer-0 窗口的应用，WindowServer 会按应用重新组织窗口顺序，无法把另一个进程的四个 panel 精确插入这些 sibling 窗口之间。

当前实现要求四个 panel 必须紧挨目标窗口。ToDesk sibling 窗口被插入 panel 与目标之间后，延迟校验会判定层级无效并隐藏全部浮层。每次拖动结束或定时纠偏又会短暂重新显示 panel，随后再次隐藏，因此表现为拖动时闪烁、静止后无浮层。

真机探针已确认：

- Chrome、CSES、系统信息等普通窗口的四个 panel 可以紧邻目标窗口；
- ToDesk `Window 16303`、`Window 16818` 的 panel 会被放到整个 ToDesk 窗口组上方；
- ToDesk sibling 窗口与目标窗口存在大面积重叠，不能简单放宽层级校验，否则 overlay 会错误显示在遮挡窗口之上。

## 目标

- 目标窗口未被覆盖的角正常显示共享边框；
- 任一角被其他窗口覆盖时，该角的 overlay 不得显示在遮挡窗口之上；
- 拖动窗口期间不显示瞬态错误层级，消除闪烁；
- 鼠标松开、应用切换、空间切换和定时纠偏后重新计算可见角；
- 普通单窗口应用继续显示完整四角浮层；
- 不使用 macOS 私有 CGS API，不要求辅助功能或输入监控权限；
- 不改变 ScreenCaptureKit、WebRTC、显示器共享及 Windows 实现。

## 方案比较

### 方案 A：按角计算 sibling 遮挡并选择性显示（采用）

在排序前读取目标窗口上方的窗口栈，识别 WindowServer 可能重新放到 panel 下方的同进程、同 layer sibling 窗口。分别计算四个 32×32 角区域是否与这些窗口相交，只显示未被遮挡的角。

优点是完全使用公开 API，普通窗口行为不变，并且不会把 overlay 绘制到真实遮挡窗口上方。代价是需要维护每个 panel 的独立可见状态。

### 方案 B：放宽为“panel 位于目标上方即可”（不采用）

该方案改动最小，可以让 ToDesk 浮层持续显示，但 panel 实际位于整个 ToDesk 窗口组上方。当 sibling 窗口遮挡目标时，overlay 仍会压在遮挡窗口上，违反既定层级要求。

### 方案 C：使用私有 CGS 窗口排序 API（不采用）

私有 API 可能实现跨进程逐窗口插入，但存在系统版本兼容、签名、审核和运行时稳定性风险，不适合插件公共实现。

## 设计

### 窗口栈快照

扩展 macOS 窗口快照，使层级判定能够获得：

- 窗口 ID；
- WindowServer layer；
- owner PID；
- AppKit 坐标系下的窗口边界；
- 窗口在 `CGWindowListCopyWindowInfo` 中的前后顺序。

只把目标窗口上方、与目标同 owner PID、同 layer 的窗口视为“应用分组 sibling”。其他应用窗口仍由正常 WindowServer 应用层级负责遮挡，不通过本地隐藏模拟。

### 四角可见性

先通过 `corner_panel_frames` 得到四个角的区域，再逐角检查 sibling 窗口边界：

- 没有 sibling 与角区域相交：该 panel 可以显示；
- 任一 sibling 与角区域相交：该 panel 保持隐藏；
- 四个角都被覆盖：浮层整体不可见，但会话仍正常运行；
- sibling 只覆盖目标内部、不覆盖边框角：四个 panel 不受影响。

矩形相交采用严格正面积判断，边界刚好接触不算遮挡。无效、非有限或空边界按 fail-closed 处理，不用于显示新的 panel。

### 排序与校验

每次刷新按以下顺序执行：

1. 读取目标几何和排序前窗口栈；
2. 计算本轮四角可见掩码；
3. 更新四个 panel 的几何；
4. 只对可见角调用相对排序，对遮挡角保持 `orderOut`；
5. 下一轮主 RunLoop 校验本轮代次。

延迟校验不再要求四个 panel 全部紧挨目标，而是验证：

- 期望显示的 panel 均已出现在目标上方且 layer 一致；
- 期望隐藏的 panel 未出现在屏幕窗口列表中；
- panel 与目标之间若存在 sibling 窗口，该 sibling 不得与该 panel 的角区域相交；
- 出现未知的跨应用窗口顺序异常时继续 fail-closed，隐藏受影响 panel。

### 事件与闪烁控制

- 收到 `LeftMouseDragged` 后立即隐藏全部 panel、暂停纠偏 Timer，并取消待校验代次；
- 拖动期间不重新排序或显示 panel；
- 收到 `LeftMouseUp` 后读取稳定窗口栈，一次性重算可见掩码并显示安全角；
- 应用激活、隐藏、终止、Space 切换、显示器变化和系统唤醒复用同一刷新流程；
- 低频纠偏只在几何、层级或可见掩码变化时执行原生更新。

这样拖动过程不会出现“先显示再校验隐藏”的一帧闪烁。

## 错误处理

- 目标窗口关闭、离开当前 Space 或边界无效：隐藏全部 panel，并返回可恢复的来源不可用错误；
- 无法读取窗口列表：隐藏全部 panel，记录结构化 debug 日志；
- 延迟校验失败：只隐藏不安全的 panel，不停止已建立的采集和发布；
- 过期校验代次、暂停、停止或正在拖动：直接忽略，不改变当前状态。

## 测试

### 纯逻辑回归测试

- 普通窗口栈：四角全部可见；
- ToDesk 模式：同 owner sibling 位于 target 上方且覆盖部分角，只隐藏被覆盖角；
- sibling 覆盖目标内部但不碰角，四角仍可见；
- 其他应用窗口位于 target 上方，不误归类为 sibling；
- 角矩形只接触边界时不算遮挡；
- 拖动隐藏、松手重算；
- 旧校验代次不能覆盖新可见掩码。

### 真机反馈环

修正 `scripts/macos-overlay-order-probe.swift` 的目标选择，排除 Window Server、无有效边界和系统状态窗口，并增加 ToDesk 指定窗口 ID 模式。探针断言：

- 普通窗口四角连续排序有效；
- ToDesk sibling 会触发应用分组；
- 与 sibling 相交的角不会显示 panel；
- 未遮挡角持续显示，等待至少一个纠偏周期后仍不消失；
- 自动移动目标窗口期间无 panel 闪烁，松手后恢复稳定掩码。

### 完整验证

- macOS 浮层策略测试；
- feature-enabled 全量测试；
- Clippy 严格检查；
- 示例应用构建；
- 格式和 diff 检查。

## 非目标

- 不保证在目标窗口完全被同应用 sibling 覆盖时仍显示边框；此时正确行为就是没有可见 overlay；
- 不对任意形状、透明区域或窗口阴影做像素级遮挡；本实现按 WindowServer 窗口边界矩形判断；
- 不把 overlay 绘制进共享视频帧；
- 不改变来源枚举和 ToDesk 窗口过滤策略。
