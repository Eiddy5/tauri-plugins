# 手动共享清晰度档位方案（已采纳）

> 方案状态：**ADOPTED（已采纳）**
>
> 采纳标记：`ADOPTED-MANUAL-CAPTURE-QUALITY-PRESETS`
>
> 日期：2026-07-13
>
> 适用平台：示例 App（Windows 已实现，macOS 沿用相同捕获尺寸契约）

## 问题背景

示例 App 原来固定把共享输出限制为最大宽度 1920，因此无论源屏幕是 2K 还是 4K，最终都只能输出最高 1080p。插件底层接受调用方传入的宽高，但界面没有暴露选择入口；同时 Windows 编码码率封顶 8 Mbps，即使直接传入 2K/4K 尺寸也无法为高分辨率文字提供匹配的质量预算。

## 解决目标

- 用户可以在开始共享前手动选择 720p、1080p、2K 或 4K；
- 720p、1080p、2K 保持 60 FPS，4K 使用实机可持续的 30 FPS，并保持源内容宽高比；
- 不放大小于所选档位的窗口，避免无意义插值；
- H.264 所需宽高自动归一为偶数；
- 高分辨率档位同步提升 Windows 硬件编码码率上限；
- 共享中锁定清晰度选择，避免界面声称已切换但底层会话仍使用旧尺寸；
- 用户选择持久化，下次启动继续使用。

## 已采纳方案

| 档位 | 最大宽度 | 典型 16:9 输出 | 帧率 | Windows 最大目标码率 |
| --- | ---: | ---: | ---: | ---: |
| 720p HD | 1280 | 1280 × 720 | 60 FPS | 约 4.4 Mbps |
| 1080p Full HD | 1920 | 1920 × 1080 | 60 FPS | 8 Mbps |
| 2K QHD | 2560 | 2560 × 1440 | 60 FPS | 约 17.7 Mbps |
| 4K UHD | 3840 | 3840 × 2160 | 30 FPS | 约 19.9 Mbps |

默认档位仍为 1080p，兼顾清晰度、GPU 压力和网络带宽。输出尺寸计算为 `min(1, presetMaxWidth / sourceWidth, presetMaxHeight / sourceHeight)`，所以横屏、竖屏都不会超过档位的完整边界框，窗口小于档位上限时保持接近原始尺寸。

编码画布在会话开始后保持固定，窗口拖动改变宽高比时，Windows GPU video processor 会重新计算居中的等比例目标区域，并用黑边填充画布剩余区域；CPU fallback 使用相同 letterbox 规则，避免把动态 resize 后的新画面拉伸到旧宽高比。

Windows 将原来分散在 Media Foundation、OpenH264 fallback 和编码 worker 中的码率计算统一为同一策略：1080p 及以下保持最高 8 Mbps，超过 1080p 后按像素数与帧率扩展，底层最高支持 32 Mbps。当前 4K30 档的最大目标约为 19.9 Mbps。REMB 仍可在每个档位的最大码率与 50% 质量下限之间调整。

## 交互约束

1. 只能在未共享时选择清晰度；
2. 开始共享后选择框禁用；
3. 更换档位需要停止并重新开始共享；
4. 预览区域在开始前显示该源的实际输出宽高与所选帧率；
5. 选择通过 `screenCapture.quality` 保存到本地存储。

## 降级与边界

- 4K 代表插件请求的最大输出尺寸，不保证所有显卡都支持 4K30 H.264 硬件编码；不支持时启动会返回明确错误，不静默伪装为 4K。
- 本机实测 4K60 只能达到约 49–51 FPS，因此采纳 4K30 作为可持续档位；需要最高运动流畅度时应选择 2K60 或 1080p60。
- 如果源本身只有 1080p，选择 2K/4K 不会放大，实际仍输出接近 1080p。
- 高档位需要更高网络预算。网络长期低于质量下限时，后续应实现运行时分辨率/帧率自适应，而不是继续降低同一分辨率的文字质量。

## 验证要求

- 前端构建必须通过；
- Rust 单元测试覆盖典型 720p60、1080p60、2K60、4K30/60 码率结果；
- 全量 Rust 测试与 Clippy 必须通过；
- Windows 示例 App 实测选择框、持久化、会话期间禁用和实际输出统计。

## Windows 实机验证

在 3840 × 2160 显示器上选择 4K 档并通过本地 WebRTC 回环播放：

| 指标 | 结果 |
| --- | --- |
| 请求输出 | 3840 × 2160 @ 30 FPS |
| 发布帧率 | 30.0 FPS |
| 捕获帧率 | 约 28 FPS，缺帧由统一 deadline cadence 补齐 |
| 实际编码目标 | 9954 kbps（REMB 处于 4K30 质量下限） |
| CPU readback | 0 |
| 编码后端 | `media-foundation-d3d11-surface` |
| Receiver Report | `total_lost=0` |
| 清晰度控件 | 会话期间禁用，重启后保留 4K 选择 |

测试同时发现旧节拍在 4K60 下会把约 44 FPS 的真实帧与续发 tick 叠加为约 67 FPS。已改为统一 deadline：每个周期只编码最新真实帧，没有新帧才续发上一 GPU surface，避免超发。

## 决策

本方案作为示例 App 的默认手动清晰度控制正式采纳，默认选中 1080p Full HD。
